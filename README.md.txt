# Relatório de Justificativa

Escolhi fazer a remoção de forma recursiva principalmente porque o código base da árvore (a inserção e os percursos) já foi construído usando recursão. Mantendo essa mesma estratégia, o código fica padronizado e muito mais limpo. Além disso, a recursão facilita muito a atualização dos ponteiros em Java. Quando um nó é deletado, o próprio retorno do método já atualiza a ligação do nó pai automaticamente, o que evita ter que criar variáveis extras para rastrear manualmente quem era o pai do nó atual, como aconteceria em uma lógica iterativa com loops.

Essa facilidade com os ponteiros ajuda bastante no caso de nós com dois filhos. Como a regra escolhida foi a do Sucessor (menor dos maiores), depois que eu encontro esse menor valor da direita e coloco ele no lugar do nó apagado, a remoção do nó sucessor original lá embaixo se resolve com apenas mais uma linha chamando o próprio método de remover de novo. Se fosse iterativo, o código ficaria gigante e muito confuso de ler.
